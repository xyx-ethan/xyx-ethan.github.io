    1. Game AI Introduction

    Game AI is fundamentally different from academic AI. The goal is not optimal play — it is creating fun, engaging player experiences. An unbeatable chess engine makes a terrible opponent in a casual game. Game AI must entertain, challenge appropriately, and sometimes intentionally lose.

    Game AI vs. Academic AI: Academic AI seeks optimal solutions to well-defined problems (e.g., minimizing loss, maximizing reward). Game AI prioritizes perceived intelligence and player enjoyment over actual optimality. A perfectly rational agent is often a poor game opponent because it removes challenge calibration, surprise, and the illusion of a believable character.

    1.1 Roles of AI in Games

    AI performs a wide range of functions beyond controlling enemies:

    • Opponent control — the most obvious role: enemies, rival teams, boss encounters
    • Player replacement — substituting for missing human players in multiplayer sessions
    • Player aids — camera control, auto-aim assist, smart targeting
    • Procedural content generation — creating levels, terrain, quests, and narrative dynamically

    1.2 Genre-Specific AI

    Different game genres demand radically different AI techniques:

    • Turn-based games — minimax search, alpha-beta pruning, Monte Carlo tree search (MCTS). Classic AI territory where look-ahead planning dominates
    • Projectile games — physics simulation combined with error distributions to create believable, not perfect, aiming
    • Platformers — simple fodder enemies with basic patrol and chase behaviors. Complexity lives in level design, not AI
    • Fighting games — adaptation to player patterns, keystroke analysis, case-based reasoning to counter repeated strategies
    • FPS (first-person shooters) — bots need perceptual models (can they see/hear the player?), aim error distributions, tactical reasoning about cover and flanking
    • RTS (real-time strategy) — pathfinding at scale, formation movement, resource management, strategic planning across multiple fronts
    • RPGs — interactive narrative systems, procedural quest generation, companion AI that assists without stealing the spotlight
    • Racing — rubber-banding (Mario Kart: slowing leaders, boosting trailers) vs. realistic simulation (Gran Turismo: modeling tire grip, racing lines)
    • Sports — team coordination, realistic athlete behavior, adapting difficulty to player skill
    • God games — indirect control where the player sets goals but AI agents decide how to achieve them (The Sims, Black & White)

    1.3 How Good Should AI Be?

    A crucial design question is how intelligent game AI should appear:

    • Learning is not always desired — in platformers, enemies should behave consistently so players can learn patterns
    • Intentional dumbing down — AI that could play perfectly is deliberately weakened to give players a chance
    • Cheating is acceptable — AI often has access to information a human player would not, as long as the player never notices
    • Distribute intelligence — split cleverness between the agent code and the environment. Pre-placed cover spots as tagged waypoints, boss arenas designed to create interesting encounters, and scripted triggers all reduce the burden on real-time AI

    1.4 Key Techniques Overview

    • Steering behaviors — low-level movement (seek, flee, arrive, pursue, evade, wander)
    • Path planning — finding routes through environments (A*, nav meshes, waypoint graphs)
    • Decision making — choosing what to do (finite state machines, behavior trees, rule-based systems)
    • Goal-oriented action planning (GOAP) — planning sequences of actions to satisfy goals
    • Procedural content generation — algorithmically creating game content (terrain, levels, quests)
    • Fuzzy logic — reasoning with imprecise categories (health is "low" rather than "below 25")
    • Case-based reasoning — solving new problems by adapting solutions from similar past situations

    2. Basic Agent Movement

    Agent movement is the foundation of all game AI. Before an agent can plan or decide, it must be able to move through the world smoothly and believably. This section covers the simplest kinematic movement algorithms.

    Steering: In game AI, steering refers to velocity-based movement where an algorithm computes a desired velocity (or acceleration) each frame to guide an agent toward a goal. Unlike direct positional teleporting, steering produces smooth trajectories because the agent's position changes continuously as a function of its velocity and the elapsed time step.

    2.1 Discrete vs. Continuous Movement

    Discrete movement operates on a grid: agents occupy cells and move one cell at a time (chess, turn-based tactics). Continuous movement uses floating-point positions with high frame-rate animation, producing smooth motion (action games, simulations). Most modern games use continuous movement.

    2.2 The Simulation Loop

    Every game frame follows the same cycle:

    1. Input — read player controls and sensor data
    2. Update — agents calculate movement, physics step, game logic
    3. Render — draw the scene
    4. Sync/Sleep — wait to maintain target frame rate

    AI movement calculations happen during the update phase. In 3D games, movement typically occurs in the XZ plane (horizontal ground plane) with Y representing the vertical axis.

    2.3 Seek Behavior

    The most fundamental movement: steer directly toward a target. Given agent position $\mathbf{B}$ and target position $\mathbf{A}$:

    1. Calculate the relative vector: $\mathbf{BA} = \mathbf{A} - \mathbf{B}$
    2. Normalize to a unit vector: $\hat{\mathbf{BA}} = \mathbf{BA} / |\mathbf{BA}|$
    3. Multiply by speed: $\mathbf{v}_{\text{agent}} = \text{speed} \cdot \hat{\mathbf{BA}}$
    $$\mathbf{v}_{\text{agent}} = \text{speed} \cdot \hat{\mathbf{BA}}$$

    2.4 Frame-Rate Independent Movement

    Never use a fixed frame time. Frame rates vary across machines and even frame-to-frame. Always measure elapsed time and scale movement accordingly:

    $$\mathbf{B'} = \mathbf{B} + \mathbf{v}_{\text{agent}} \cdot \Delta t$$

    In Unity, use Time.deltaTime. In a custom engine, measure wall-clock time between frames. Without delta-time scaling, agents move faster on faster machines — a classic bug.

    # Frame-rate independent movement
def update(agent, target, speed, dt):
    direction = target.position - agent.position
    distance = direction.magnitude()
    if distance > 0:
        direction = direction / distance  # normalize
    agent.velocity = direction * speed
    agent.position += agent.velocity * dt

    2.5 Orientation

    Agents should face the direction they are moving. Convert a velocity vector to an angle using atan2:

    $$\theta = \text{atan2}(v_y, v_x)$$

    The atan2 function handles all four quadrants correctly, unlike atan which only covers two.

    2.6 Kinematic Algorithms

    Kinematic algorithms set velocity directly (no acceleration). They are simple but produce instant direction changes:

    • Seek — move toward target at full speed
    • Flee — reverse the seek direction to move away from target
    • Arrive — like seek but with a slowdown radius and capture radius. Inside the slowdown radius, target speed decreases linearly. A time-to-target constant (~0.25s) controls deceleration smoothness
    • Wander — random angular offset applied to current heading, biased toward zero for gentle curves rather than jittery zigzags
    • Path Following — visit a sequence of waypoints using seek with a capture radius to trigger advancing to the next waypoint
    def kinematic_arrive(agent, target, max_speed, slow_radius, capture_radius):
    direction = target - agent.position
    distance = direction.magnitude()

    if distance < capture_radius:
        agent.velocity = Vector2(0, 0)
        return

    if distance < slow_radius:
        # Linear interpolation: slow down as we approach
        target_speed = max_speed * (distance / slow_radius)
    else:
        target_speed = max_speed

    # Set velocity toward target at computed speed
    agent.velocity = direction.normalized() * target_speed
    agent.orientation = atan2(agent.velocity.y, agent.velocity.x)

    2.7 Statics vs. Kinematics

    Statics vs. kinematics: Position and orientation are statics — they describe where an agent is. Velocity and acceleration are kinematics — they describe how an agent moves. Kinematic algorithms directly set velocity; dynamic (steering) algorithms set acceleration and let velocity evolve.

    3. Computational Geometry

    Robust geometry computations are essential for game AI — from collision detection to path planning to nav mesh construction. Floating-point arithmetic introduces subtle errors that can break algorithms. This section covers the mathematical foundations and practical techniques for handling them.

    3.1 Floating-Point Problems

    IEEE 754 32-bit floats have limited precision (~7 decimal digits). Problems include:

    • Rounding errors — not all decimal values are exactly representable in binary
    • Varying precision — the gap between representable values grows as the exponent increases. Near 1.0 the gap is ~$10^{-7}$; near 1000.0 the gap is ~$10^{-4}$
    • Catastrophic cancellation — subtracting nearly equal numbers loses significant digits

    3.2 Epsilon Testing

    Never compare floats with ==. Use an epsilon tolerance:

    # Bad: fails due to floating-point rounding
if v == target:
    print("arrived")

# Good: epsilon comparison
epsilon = 1e-6
if abs(v - target) < epsilon:
    print("arrived")

    3.3 Integer Representation

    For critical geometry operations (intersection tests, winding number computations), convert to integers by multiplying by a scale factor:

    # Convert float coordinates to integer representation
SCALE = 1000
ix = int(x * SCALE)
iy = int(y * SCALE)
# Integer arithmetic is exact (no rounding errors)

    Watch for overflow — intermediate products of large integers can exceed the representable range. Reorder equation terms to minimize intermediate values. Perform conversion during baking/preprocessing, not at runtime, to avoid the performance cost.

    3.4 Twice-Area and the Left Predicate

    The signed twice-area of triangle $ABC$ is a fundamental building block for many geometry tests:

    $$\text{Area2}(A, B, C) = (b_x - a_x)(c_y - a_y) - (c_x - a_x)(b_y - a_y)$$

    This value encodes the orientation of the triangle:

    • Left predicate: $\text{Area2} > 0$ means $C$ is to the left of the directed line from $A$ to $B$
    • Left-on: $\text{Area2} \geq 0$ (includes collinear points)
    • Collinear: $\text{Area2} = 0$ means $A$, $B$, $C$ lie on the same line
    def area2(a, b, c):
    """Signed twice-area of triangle ABC."""
    return (b[0] - a[0]) * (c[1] - a[1]) - (c[0] - a[0]) * (b[1] - a[1])

def left(a, b, c):
    """True if C is strictly left of directed line AB."""
    return area2(a, b, c) > 0

def left_on(a, b, c):
    """True if C is left of or on directed line AB."""
    return area2(a, b, c) >= 0

def collinear(a, b, c):
    """True if A, B, C are collinear."""
    return area2(a, b, c) == 0

    3.5 Line Segment Intersection

    Two segments $\overline{AB}$ and $\overline{CD}$ intersect if and only if:

    • $C$ and $D$ are on opposite sides of line $AB$ (one left, one right), and
    • $A$ and $B$ are on opposite sides of line $CD$

    This is tested using the left predicate with the triangle area sign test.

    Proper intersection means the segments cross at interior points (neither endpoint touches the other segment). Improper intersection means an endpoint lies on the other segment.

    3.6 Point-to-Line-Segment Distance

    To find the closest point on segment $\overline{AB}$ to point $P$:

    1. Project $P$ onto the infinite line through $A$ and $B$ using the normalized parameter $t$
    2. If $t \leq 0$, the closest point is $A$; if $t \geq 1$, the closest point is $B$; otherwise it is $A + t \cdot (B - A)$
    3. Avoid square roots by working with squared distances whenever possible
    def point_segment_distance_sq(p, a, b):
    """Squared distance from point P to segment AB."""
    ab = (b[0] - a[0], b[1] - a[1])
    ap = (p[0] - a[0], p[1] - a[1])
    ab_sq = ab[0]**2 + ab[1]**2

    if ab_sq == 0:
        return ap[0]**2 + ap[1]**2  # A == B

    t = (ap[0] * ab[0] + ap[1] * ab[1]) / ab_sq
    t = max(0.0, min(1.0, t))  # clamp to segment

    closest = (a[0] + t * ab[0], a[1] + t * ab[1])
    dx = p[0] - closest[0]
    dy = p[1] - closest[1]
    return dx * dx + dy * dy
    Performance tip: Square root is expensive. For comparisons (is point closer to A or B?), compare squared distances. Only take the square root when you need the actual distance value for display or physics calculations.
    Convex Hull: The convex hull of a point set is the smallest convex polygon that contains all the points, analogous to stretching a rubber band around pins on a board. It is a foundational primitive in computational geometry, used for bounding-volume collision tests, nav mesh construction, and shape simplification. Common algorithms include Graham scan ($O(n \log n)$) and gift wrapping ($O(nh)$).
    GJK Algorithm: The Gilbert-Johnson-Keerthi algorithm determines whether two convex shapes overlap by searching for the origin within their Minkowski difference. Instead of computing the full Minkowski difference, GJK iteratively builds a simplex (line, triangle, tetrahedron) using support functions, making it both efficient and general-purpose. It is the standard narrow-phase collision detection method in physics engines such as Box2D and Bullet.

    4. Path Planning

    Path planning solves the problem that simple seek behavior cannot: finding a route around obstacles. Seek drives directly toward the target and gets stuck in local minima when obstacles block the direct path. Planning considers multiple possible routes and selects the best one.

    4.1 Discretization

    The continuous game world must be converted into a discrete graph (nodes + edges) that search algorithms can operate on. Common approaches:

    • Grid lattice — regular grid overlaid on the world
    • Waypoints — hand-placed or algorithmically generated nodes
    • Navigation mesh (nav mesh) — convex polygons covering walkable areas
    • Voxels — 3D grid cells for flying or swimming agents

    4.2 Planning as Search

    Planning means considering possibilities without committing. A search problem is defined by:

    • States — configurations the agent can be in (positions, grid cells, polygon regions)
    • Operators — actions that transition between states (move north, traverse edge)
    • Start state and goal state

    4.3 Search Algorithms

    Breadth-First Search (BFS) — expand nodes level by level using a FIFO queue. Complete and optimal for equal-weight edges. Time and space complexity: $O(b^d)$ where $b$ is the branching factor and $d$ is the depth.

    Depth-First Search (DFS) — expand the deepest node first using a LIFO stack. Not optimal, but useful for maze generation when combined with random neighbor shuffling.

    Dijkstra / Uniform-Cost Search (UCS) — like BFS but expands the node with the lowest cost-so-far. Uses a priority queue. Optimal for positive edge weights.

    Greedy Best-First Search — expands the node closest to the goal according to a heuristic $h(n)$. Fast but not optimal — can get stuck behind obstacles.

    A* — the gold standard for game pathfinding. Combines cost-so-far with a heuristic estimate:

    $$f(n) = g(n) + h(n)$$

    where $g(n)$ is the cost from start to $n$, and $h(n)$ is the estimated cost from $n$ to the goal. A* is optimal if $h$ is admissible (never overestimates the true cost). Uses a priority queue — a binary heap gives $O(\log n)$ insert and extract-min.

    A* Algorithm: A* is a best-first graph search that expands the node minimizing $f(n) = g(n) + h(n)$, where $g(n)$ is the actual cost from the start and $h(n)$ is a heuristic estimate to the goal. It is guaranteed to find the shortest path when the heuristic is admissible (never overestimates) and consistent (satisfies the triangle inequality). A* is the most widely used pathfinding algorithm in games due to this optimality guarantee combined with efficient heuristic-guided pruning. 
    def a_star(start, goal, graph, heuristic):
    open_set = PriorityQueue()
    open_set.put((0, start))
    came_from = {start: None}
    cost_so_far = {start: 0}

    while not open_set.empty():
        _, current = open_set.get()

        if current == goal:
            return reconstruct_path(came_from, goal)

        for neighbor, edge_cost in graph.neighbors(current):
            new_cost = cost_so_far[current] + edge_cost

            if neighbor not in cost_so_far or new_cost < cost_so_far[neighbor]:
                cost_so_far[neighbor] = new_cost
                priority = new_cost + heuristic(neighbor, goal)
                open_set.put((priority, neighbor))
                came_from[neighbor] = current

    return None  # no path found

def reconstruct_path(came_from, goal):
    path = []
    current = goal
    while current is not None:
        path.append(current)
        current = came_from[current]
    path.reverse()
    return path

    4.4 Admissible Heuristics

    • Euclidean distance — always admissible (straight-line is always the shortest), best for 8-way movement or any-angle movement
    • Manhattan distance — admissible for 4-way grid movement: $h = |x_1 - x_2| + |y_1 - y_2|$
    • Cluster heuristic — precompute distances between room centers for environments with rooms and corridors

    4.5 Node Records

    Each node in the search maintains a record containing:

    • From-node — which node we came from (for path reconstruction)
    • Cost-so-far $g(n)$ — total cost from start to this node
    • Heuristic estimate $h(n)$ — estimated cost to goal

    An open set holds nodes to be explored; a closed set holds already-explored nodes.

    4.6 Grid Lattice

    A regular grid overlaid on the game world. Each cell is either traversable or blocked. Advantages: easy to implement, natural for tile-based games. Disadvantages: dense representation wastes memory in open areas, paths follow grid lines producing unnatural zigzag motion (needs string pulling to smooth).

    4.7 Path Network (Waypoint Graph)

    A sparse set of nodes placed at strategic locations (corners, doorways, intersections) connected by edges. Much smaller than a grid, but requires robust local movement between waypoints. Nodes can be placed at points of visibility or generated dynamically (breadcrumb approach — like the Reaper bot for Quake, which drops waypoints as it explores).

    4.8 Navigation Mesh

    A nav mesh covers the walkable area with convex polygons. Polygon adjacency defines graph edges. Key benefits:

    • Efficient representation — a few polygons replace thousands of grid cells
    • Simplified movement — any point inside a convex polygon has line-of-sight to any other point in the same polygon
    • Supports string pulling — the Simple Stupid Funnel Algorithm tightens paths through the polygon channel
    Navigation Mesh (Nav Mesh): A navigation mesh is a spatial representation of walkable area composed of convex polygons whose adjacency graph defines traversable connections. Because any two points inside a single convex polygon have unobstructed line-of-sight, agents can move freely within each polygon without additional collision checks. Nav meshes are the industry-standard world representation in engines like Unity and Unreal, replacing dense grids with a compact polygon set.

    4.9 Nav Mesh Generation

    Several approaches exist for generating navigation meshes:

    • Greedy triangulation — $O(n^4)$ brute force: try all possible diagonals, keep those that don't intersect existing edges
    • Delaunay triangulation — maximizes minimum angles, avoids thin triangles
    • Recast — voxel-based approach used in industry (Unity, Unreal): voxelize geometry, identify walkable surfaces, build polygons

    Post-processing steps include triangle merging (combine adjacent coplanar triangles into larger convex polygons), adjacency detection via hash tables (shared edges between polygons), and expanded geometry (offset obstacles by the agent's radius so the point-agent doesn't clip through walls).

    4.10 String Pulling and Path Refinement

    Raw paths from graph search follow edges between polygon centers, producing zigzag routes. String pulling tightens the path by cutting corners through the polygon channel. The funnel algorithm maintains a funnel (two boundary edges) as it walks the channel, narrowing it at each portal (shared edge between polygons).

    4.11 Quad Trees

    A variable-resolution spatial subdivision. Large empty areas use a single node; cluttered areas subdivide recursively. Advantages over grids: memory-efficient for sparse environments, fast edits for dynamic obstacles (only rebuild affected quadrants). Commonly used for RTS games with large open maps.

    • Grid — simple, dense, good for tile games, needs string pulling
    • Path network — sparse, hand-placed or auto-generated, needs local navigation
    • Nav mesh — industry standard, efficient, supports smooth paths natively
    • Quad tree — variable resolution, good for dynamic environments
    • A* — optimal with admissible heuristic, priority queue is essential for performance

    5. Steering Behaviors

    Steering behaviors extend kinematic movement by using acceleration instead of directly setting velocity. This produces smooth, natural-looking motion with gradual turns and speed changes, unlike kinematic algorithms that produce instant direction snaps.

    5.1 Acceleration-Based Movement

    The update equations for dynamic (steering) movement:

    $$\mathbf{p'} = \mathbf{p} + \mathbf{v} \cdot \Delta t$$ $$\mathbf{v'} = \mathbf{v} + \mathbf{a} \cdot \Delta t$$

    After updating velocity, clamp speed to the agent's maximum: if $|\mathbf{v'}| > v_{\max}$, then $\mathbf{v'} = v_{\max} \cdot \hat{\mathbf{v'}}$.

    def steering_update(agent, acceleration, dt, max_speed):
    # Update position
    agent.position += agent.velocity * dt
    # Update velocity
    agent.velocity += acceleration * dt
    # Clamp to max speed
    speed = agent.velocity.magnitude()
    if speed > max_speed:
        agent.velocity = agent.velocity.normalized() * max_speed

    5.2 Fundamental Steering Behaviors

    • Seek — compute desired velocity toward target, output acceleration = desired - current velocity
    • Flee — reverse of seek: accelerate away from the target
    • Arrive — like seek, but with a slowdown radius. Inside the radius, desired speed decreases via linear interpolation. Outside, use maximum speed
    • Align — match a target orientation using angular acceleration, with a slowdown region for smooth stopping

    5.3 Minimum Turning Radius

    An agent with maximum speed $v_{\max}$ and maximum acceleration $a_{\max}$ has a minimum turning radius:

    $$R = \frac{v_{\max}^2}{a_{\max}}$$

    If a target is inside this radius, the agent cannot turn sharply enough and will orbit the target instead of reaching it. This is a key limitation of pure steering — arrive behavior solves it by slowing down within the slowdown radius.

    5.4 Turn-First Steering

    An alternative approach that solves the orbiting problem:

    1. Lock orientation to face the velocity direction
    2. When turning, project the previous velocity onto the new facing direction — this naturally decelerates the agent during sharp turns
    3. Use variable forward acceleration based on the angle to the target: full acceleration when facing it, reduced when turning

    5.5 Delegated Behaviors

    Higher-level behaviors that compute a target and then delegate to a fundamental behavior:

    • Pursue — predict the target's future position using $t_{\text{est}} = \text{distance} / \text{speed}$, then seek that predicted position. Assumes constant target velocity
    • Evade — predict the pursuer's future position, then flee from it
    • Wander — place a circle in front of the agent with a rotating target point on its circumference. Produces smooth, organic-looking random movement
    • Path Following — find the nearest point on the path, then seek a point ahead of that position on the path
    • Separation — force field that pushes agents apart. Can use linear or inverse-square falloff depending on desired behavior
    • Collision Avoidance — cone test (will our velocity cones overlap?) or closest-approach calculation between agent trajectories
    • Obstacle Avoidance — cast a ray ahead along the velocity direction. If it hits an obstacle, steer away. Use whisker rays at angles for wider detection. Corner traps (concave environments where whiskers detect obstacles on both sides) can be solved with an adaptive fan angle
    def pursue(agent, target, max_speed):
    """Predict target's future position and seek it."""
    displacement = target.position - agent.position
    distance = displacement.magnitude()

    # Estimate time to intercept
    speed = agent.velocity.magnitude()
    if speed <= 0:
        t_predict = 1.0
    else:
        t_predict = distance / speed

    # Predict future position assuming constant velocity
    future_pos = target.position + target.velocity * t_predict

    # Delegate to seek
    return seek(agent, future_pos, max_speed)

    5.6 Blended Steering

    Combine multiple behaviors by weighted summation of their output accelerations:

    $$\mathbf{a}_{\text{total}} = w_1 \mathbf{a}_{\text{seek}} + w_2 \mathbf{a}_{\text{avoid}} + w_3 \mathbf{a}_{\text{separate}} + \cdots$$

    Priority blending evaluates behavior sets in priority order and uses the first set that produces an acceleration above a threshold. This prevents low-priority behaviors (wandering) from interfering with high-priority ones (obstacle avoidance).

    Weighted-Sum Steering Combination: Multiple steering behaviors are combined by computing a weighted sum of their individual acceleration outputs: $\mathbf{a}_{\text{total}} = \sum_i w_i \mathbf{a}_i$. The weights encode behavioral priorities (e.g., obstacle avoidance weighted heavily, wander weighted lightly). This linear blending is simple and efficient but can produce degenerate equilibria when opposing forces cancel out, which is why priority-based blending is often layered on top.
    Equilibrium problems: Blended steering can produce pathological cases. Unstable equilibria occur when forces balance perfectly (e.g., fleeing exactly between two threats), sending the agent in a random direction. Stable equilibria trap the agent (e.g., equal-weight seek and obstacle avoidance cancelling out near a wall). Priority blending mitigates but doesn't eliminate these issues.

    5.7 Performance Envelope

    In animation-driven games, the agent's animations define its movement capabilities. The performance envelope is the set of (speed, turn rate) pairs that the animation system can produce:

    • Root motion drives translation and rotation from animation data
    • Lookup tables map desired movement parameters to animation blend weights
    • Different agents with different animation sets have different performance envelopes, creating unique movement personalities

    5.8 Creating Unique Agents

    • Vary weights on blended behaviors (more aggressive = higher seek weight, more cautious = higher separation weight)
    • Adjust parameters (slowdown radius, maximum speed, acceleration limits)
    • Change priorities in priority blending (which behaviors dominate)
    • Assign different performance envelopes (animation-driven movement constraints)

    6. Ballistic Projectile Targeting

    Ballistic targeting is the problem of computing the initial velocity for a projectile (a thrown grenade, kicked ball, or fired arrow) so that it hits a target. The key constraint is that the projectile is ballistic — once launched, no mid-flight corrections are possible. The AI gets one chance to set the velocity correctly.

    Ballistic Projectile: A ballistic projectile is an object that, once launched, follows a parabolic trajectory determined solely by its initial velocity and gravitational acceleration, with no in-flight steering or thrust. Its position at time $t$ is given by $\mathbf{p}(t) = \mathbf{p}_0 + \mathbf{v}_0 t + \frac{1}{2}\mathbf{g}t^2$. Computing the required launch velocity to hit a target reduces to solving a quartic (or equivalent quadratic in $t^2$) equation, yielding zero, one, or two valid trajectories.

    6.1 Static Target, No Gravity

    The trivial case: compute the direction to the target, normalize, and multiply by projectile speed. This is just the seek calculation applied to a projectile.

    6.2 Static 3D Target with Gravity

    The projectile follows a parabolic arc under gravity. Its position at time $t$ is:

    $$\mathbf{p}(t) = \mathbf{p}_0 + \mathbf{v} t + \frac{1}{2}\mathbf{g} t^2$$

    where $\mathbf{g} = (0, -9.8, 0)$ m/s$^2$. We want to find a unit direction $\hat{\mathbf{u}}$ such that $\mathbf{v} = s \cdot \hat{\mathbf{u}}$ hits the target at time $t_c$.

    Substituting into the constraint $|\hat{\mathbf{u}}| = 1$ produces a quartic equation in $t_c^2$. Using the substitution $x = t_c^2$, this reduces to a quadratic:

    1. Check the radicand (discriminant) — if negative, no solution exists (target is out of range)
    2. Solve for $x = t_c^2$, discard negative values
    3. Take $t_c = \sqrt{x}$, pick the smallest valid $t_c$ (quickest arrival)
    4. Back-substitute to find $\hat{\mathbf{u}}$
    def solve_ballistic_static(p0, target, speed, gravity=-9.8):
    """Solve for launch velocity to hit a static target under gravity."""
    delta = target - p0
    dx, dy, dz = delta.x, delta.y, delta.z

    # Horizontal distance squared
    h_dist_sq = dx**2 + dz**2

    s2 = speed * speed
    s4 = s2 * s2
    g = abs(gravity)

    # Quadratic in t^2: coefficients
    a = g * g / 4.0
    b = -(s2 + g * dy)
    c = h_dist_sq + dy * dy

    discriminant = b * b - 4 * a * c
    if discriminant < 0:
        return None  # target out of range

    sqrt_disc = math.sqrt(discriminant)
    t_sq_1 = (-b + sqrt_disc) / (2 * a)
    t_sq_2 = (-b - sqrt_disc) / (2 * a)

    # Pick smallest positive t
    t_sq = None
    for candidate in [t_sq_2, t_sq_1]:
        if candidate > 0:
            t_sq = candidate
            break
    if t_sq is None:
        return None

    tc = math.sqrt(t_sq)

    # Compute launch velocity components
    vx = dx / tc
    vy = dy / tc - 0.5 * gravity * tc
    vz = dz / tc

    return Vector3(vx, vy, vz)

    6.3 Moving Target (Law of Cosines Approach)

    For a moving target in 2D, set up a triangle with three edges:

    • $a$ = distance from agent to target's current position
    • $b$ = target speed $\times$ $t$ (how far the target moves)
    • $c$ = projectile speed $\times$ $t$ (how far the projectile travels)

    Apply the law of cosines and the dot product to find the angle between the target's velocity and the line connecting target to agent. This yields a quadratic in $t$. Solve for the smallest positive root, then compute the required launch velocity.

    For 3D, project the problem to the XZ plane. The Y component may require a holdback trick: reserve some of the projectile's speed budget for the vertical component, reducing the horizontal speed used in the 2D solve.

    6.4 Iterative Approach

    A simpler alternative that works well in practice:

    1. Solve for a static target at the target's current position → get time $t_1$
    2. Use $t_1$ to predict where the target will be at that time (kinematic equation)
    3. Solve for the static target at that predicted position → get time $t_2$
    4. Repeat 3–6 iterations until $t$ converges

    This is easy to implement and converges quickly for most game scenarios.

    6.5 Practical Considerations

    Practical heuristics: Throwing characters have an animation wind-up delay (~0.25 seconds) that must be accounted for in timing. Good heuristics for when to throw: target is at maximum speed (predictable trajectory), target is not turning (constant velocity assumption holds), and a raycast along the parabolic arc doesn't hit obstacles. Approximate the parabolic arc with a 2-segment raycast for a cheap obstacle check.

    7. Procedural Content Generation (Terrain)

    Procedural content generation (PCG) creates game content algorithmically rather than by hand. For terrain, the goal is to produce natural-looking landscapes — mountains, valleys, craters, coastlines — that are varied, controllable, and efficiently generated.

    7.1 Perlin Noise

    Perlin noise is a gradient noise function that produces smooth, continuous random values. Unlike white noise (completely random per pixel), Perlin noise has spatial coherence — nearby points have similar values, producing natural-looking bumps and hills.

    Two key parameters control the output:

    • Frequency (scaler) — how rapidly the noise varies across space. High frequency = small, detailed features. Low frequency = large, sweeping features
    • Amplitude (maximum value) — the height range of the noise output
    Perlin Noise: Perlin noise is a gradient noise function that assigns random gradient vectors to integer lattice points and interpolates smoothly between them, producing spatially coherent pseudo-random values. Unlike white noise, neighboring sample points yield similar values, creating natural-looking continuous variation. It is the foundational building block for procedural terrain, cloud textures, and organic surface detail in games and film.

    7.2 Layered Noise (Octaves)

    A single layer of Perlin noise produces boring, uniform bumps. Interesting terrain requires detail at multiple scales. Layered noise combines octaves with diminishing amplitude:

    def layered_noise(x, y, octaves=6, persistence=0.5):
    """Generate layered Perlin noise with multiple octaves."""
    total = 0.0
    frequency = 1.0
    amplitude = 1.0
    max_value = 0.0

    for i in range(octaves):
        total += perlin_noise(x * frequency, y * frequency) * amplitude
        max_value += amplitude
        amplitude *= persistence  # each octave has less influence
        frequency *= 2.0          # each octave is higher frequency

    return total / max_value  # normalize to [0, 1]

    The first octave provides the broad terrain shape (continents, mountain ranges). Each subsequent octave adds finer detail (ridges, erosion patterns, pebbles) with decreasing influence.

    Octave Layering (Fractal Brownian Motion): A single Perlin noise pass produces uniform, featureless bumps. Octave layering sums multiple noise samples at increasing frequency and decreasing amplitude, mimicking how natural terrain exhibits self-similar detail at many scales. Each octave doubles the frequency (lacunarity) and halves the amplitude (persistence), so the first octave sets the broad landscape while later octaves add ridges, rocks, and erosion detail without overpowering the large-scale shape.

    7.3 Hierarchical Rules

    Complex terrain is built from a tree of noise generators, each contributing a different terrain feature. The process for each node:

    1. Generate own noise — produce a height map from Perlin noise with this node's parameters
    2. Process children — recursively evaluate child nodes
    3. Combine with parent — merge child output with parent using a combiner operation

    Available combiner operations:

    • Add — layer features on top of the base terrain
    • Subtract — carve out valleys, craters, riverbeds
    • Multiply — modulate one terrain by another (e.g., mask mountains to only appear in certain regions)
    • Min / Max — take the lower or higher of two terrain values
    • Normalize — rescale values to a target range
    • Replace — completely override parent values

    7.4 Mapping Functions

    Mapping functions transform raw noise values into useful terrain features by reshaping the distribution:

    Trapezoid function — fast, defined by 4 parameters that control ramp-up, plateau, and ramp-down regions. Use to isolate specific height ranges (e.g., only values between 0.6 and 0.8 become mountain peaks):

    def trapezoid(value, a, b, c, d):
    """Trapezoid mapping: ramp up [a,b], plateau [b,c], ramp down [c,d]."""
    if value <= a or value >= d:
        return 0.0
    elif value < b:
        return (value - a) / (b - a)  # ramp up
    elif value <= c:
        return 1.0                     # plateau
    else:
        return (d - value) / (d - c)   # ramp down

    Curve editor — smooth spline-based mapping with arbitrary control points. More expressive than trapezoid but slower to evaluate. Use when trapezoid approximation is too coarse.

    7.5 Practical Workflow

    1. Start with base terrain using low-frequency Perlin noise for overall shape
    2. Add child nodes for biomes: mountains, plains, ocean floor, each with appropriate frequency/amplitude
    3. Apply trapezoid functions to isolate features (mountain peaks above a threshold, craters below another)
    4. Use combiners to merge: add mountains onto base, subtract craters, multiply with moisture maps for biome selection
    5. Refine with curve editors where trapezoid approximation is too coarse
    6. Layer high-frequency octaves for fine detail (erosion, rocks, vegetation placement)

    8.1 Grid Navigation

    Discretize the continuous game world using a grid lattice. For each cell, perform an intersection test against all obstacles to determine whether the cell is traversable or blocked. The agent then uses A* on this grid to find paths.

    def create_grid(world_bounds, cell_size, obstacles):
    """Build a traversability grid for the game world."""
    cols = int(world_bounds.width / cell_size)
    rows = int(world_bounds.height / cell_size)
    grid = [[True] * cols for _ in range(rows)]

    for r in range(rows):
        for c in range(cols):
            cell_rect = get_cell_bounds(r, c, cell_size)
            for obstacle in obstacles:
                if intersects(cell_rect, obstacle):
                    grid[r][c] = False
                    break
    return grid

    8.2 Path Network

    Connect hand-placed nodes with edges to form a path network. An edge is valid only if a corridor of width equal to twice the agent radius is clear of obstacles along its entire length. This ensures the agent can traverse the edge without clipping through geometry.

    def is_valid_edge(node_a, node_b, agent_radius, obstacles):
    """Check if the corridor between two nodes is obstacle-free."""
    direction = (node_b - node_a).normalized()
    # Perpendicular offset for corridor width
    perp = Vector2(-direction.y, direction.x) * agent_radius

    # Check four corridor boundary segments
    corners = [
        node_a + perp, node_a - perp,
        node_b + perp, node_b - perp
    ]
    for obstacle in obstacles:
        if obstacle_intersects_corridor(corners, obstacle):
            return False
    return True

    8.3 Nav Mesh

    Build a navigation mesh using greedy triangulation:

    • Expanded geometry — offset all obstacle boundaries outward by the agent radius
    • Greedy triangulation — iteratively add diagonals that don't intersect existing edges, with an adjacency-blocking check to prevent non-navigable triangulations
    • Triangle merging — combine adjacent triangles into larger convex polygons where possible
    • Adjacency detection — use hash tables on shared vertex pairs to efficiently find which polygons are neighbors

    8.4 PCG Terrain

    Generate terrain using a declarative rule hierarchy:

    • Define a tree of Perlin noise generators with frequency, amplitude, and octave parameters
    • Apply mapping functions (trapezoid or curve) to shape raw noise into terrain features
    • Combine nodes using add, subtract, multiply, min, max, normalize, or replace operations
    • The system evaluates the tree for each terrain sample point, producing a final height value
    Design philosophy: Each assignment adds a layer of sophistication. Grid navigation is the simplest discretization; path networks are sparser but require careful edge validation; nav meshes are the most efficient but complex to generate. PCG terrain shifts from consuming worlds to creating them, tying together geometry and noise theory.

    9. Coordinated Movement

    9.1 Flocking — Boids Model

    Craig Reynolds' Boids (1987) simulates flocking with three simple steering rules applied independently to each agent:

    • Separation — steer away from neighbors that are too close. Compute a repulsion vector as the sum of displacement vectors from each nearby neighbor, weighted inversely by distance: $\mathbf{f}_{\text{sep}} = \sum_{i} \frac{\mathbf{p} - \mathbf{p}_i}{\|\mathbf{p} - \mathbf{p}_i\|^2}$
    • Cohesion — steer toward the centroid of the local neighborhood: $\mathbf{f}_{\text{coh}} = \frac{1}{N}\sum_{i}\mathbf{p}_i - \mathbf{p}$
    • Alignment — match the average heading/velocity of neighbors: $\mathbf{f}_{\text{ali}} = \frac{1}{N}\sum_{i}\mathbf{v}_i - \mathbf{v}$

    Each force is weighted and summed to produce a final steering vector. Tuning these weights dramatically changes emergent behavior — high separation creates loose swarms, high cohesion creates tight clusters, high alignment creates orderly flocks.

    Boids Model: The Boids model (Reynolds, 1987) is an agent-based flocking simulation in which each agent independently applies three local steering rules: separation (avoid crowding nearby neighbors), cohesion (steer toward the average position of neighbors), and alignment (match the average heading of neighbors). Despite having no global controller or leader, these three simple rules produce complex, lifelike emergent group behaviors such as flocking birds, schooling fish, and herding animals.
    Neighborhood definition: Each boid considers only neighbors within a fixed radius and within a forward-facing vision cone (typically 270°). This prevents agents from reacting to entities directly behind them, producing more natural movement.

    9.2 Spatial Partitioning — Bin Lattice

    Naive neighbor queries are $O(n^2)$. A bin lattice (uniform spatial grid) reduces this to $O(n)$ expected time:

    • Divide the world into a grid of cells, each at least as wide as the neighbor-detection radius
    • Each frame, assign every agent to its cell in $O(1)$
    • For neighbor queries, check only the agent's cell and the 8 surrounding cells (2D) or 26 surrounding cells (3D)
    • Works best when agents are roughly uniformly distributed; degenerates if all agents cluster in one cell
    # Bin lattice spatial partitioning
class BinLattice:
    def __init__(self, world_size, cell_size):
        self.cell_size = cell_size
        self.cols = int(world_size[0] / cell_size) + 1
        self.rows = int(world_size[1] / cell_size) + 1
        self.grid = [[] for _ in range(self.cols * self.rows)]

    def clear(self):
        for cell in self.grid:
            cell.clear()

    def insert(self, agent):
        cx = int(agent.pos.x / self.cell_size)
        cy = int(agent.pos.y / self.cell_size)
        self.grid[cy * self.cols + cx].append(agent)

    def get_neighbors(self, agent, radius):
        cx = int(agent.pos.x / self.cell_size)
        cy = int(agent.pos.y / self.cell_size)
        neighbors = []
        for dy in range(-1, 2):
            for dx in range(-1, 2):
                nx, ny = cx + dx, cy + dy
                if 0 <= nx < self.cols and 0 <= ny < self.rows:
                    for other in self.grid[ny * self.cols + nx]:
                        if other is not agent:
                            dist = (other.pos - agent.pos).length()
                            if dist < radius:
                                neighbors.append(other)
        return neighbors

    9.3 Formations

    Formations arrange agents into geometric patterns while moving as a group:

    • Anchor point — a virtual leader that follows the desired path; each agent maintains a fixed offset relative to the anchor
    • Slot assignment — assign agents to formation slots using a cost matrix (distance to each slot), solved greedily or with the Hungarian algorithm for optimal assignment
    • Forming up — agents steer toward their assigned slot positions; formation movement begins only when all agents are within tolerance of their slots
    • Wheeling — when the formation turns, the anchor traces the turn arc. Inner agents slow down and outer agents speed up to maintain spacing: $v_i = v_{\text{anchor}} \cdot \frac{r_i}{r_{\text{anchor}}}$ where $r_i$ is the agent's turn radius

    9.4 Obstacle Avoidance in Formations

    When a formation encounters obstacles, several strategies apply:

    • Soft constraints — agents can drift from their slot positions to avoid obstacles, then return when clear
    • Formation splitting — the formation temporarily breaks into sub-groups that pass on either side of the obstacle, then reforms
    • Dynamic reshaping — switch formation type (e.g., from wedge to column) to fit through narrow passages

    9.5 Movement Lines & Priority-Based Collision Resolution

    For large groups, a movement line approach defines a spline or polyline path that agents follow in order:

    • A leader agent plans the path; followers maintain spacing along the same path curve
    • Priority-based resolution — assign priority by formation slot order. Higher-priority agents move first; lower-priority agents treat higher-priority ones as moving obstacles and adjust accordingly
    • This avoids deadlocks where two agents block each other — the lower-priority agent always yields
    • Boids produce emergent flocking from three local rules: separation, cohesion, alignment
    • Bin lattice spatial partitioning makes neighbor queries O(n) instead of O(n²)
    • Formations use an anchor point with per-agent offsets; wheeling adjusts speed by turn radius
    • Priority-based collision resolution prevents deadlocks in group movement

    10. Perception & Fairness

    10.1 Why Perception Matters

    Omniscient AI that knows everything feels unfair to players. Perception systems limit what AI agents can sense, creating believable behavior and fair gameplay. The key principle: AI should only act on information it could reasonably perceive.

    10.2 Visual Perception

    Sight is modeled with multiple constraints:

    • Range limit — maximum detection distance, often scaled by target size: $d_{\max} = d_{\text{base}} \cdot \text{targetScale}$
    • Vision cone — a forward-facing cone defined by half-angle $\theta$. Test: $\cos^{-1}\left(\frac{\mathbf{forward} \cdot \mathbf{toTarget}}{|\mathbf{forward}||\mathbf{toTarget}|}\right) < \theta$
    • Occlusion testing — cast a ray from the agent's eye to the target. If the ray hits geometry before reaching the target, the target is occluded (not visible). Use multiple rays to different body parts for partial visibility
    • Foveal vs. peripheral vision — inner cone (narrow, high acuity) detects details and identifies targets; outer cone (wide, low acuity) detects movement but not identity
    import math

def can_see(agent, target, max_range, fov_half_angle, obstacles):
    """Check if agent can see target with range, FOV, and occlusion."""
    to_target = target.pos - agent.pos
    distance = to_target.length()

    # Range check
    if distance > max_range:
        return False

    # FOV check
    to_target_norm = to_target.normalize()
    dot = agent.forward.dot(to_target_norm)
    angle = math.acos(max(-1, min(1, dot)))
    if angle > fov_half_angle:
        return False

    # Occlusion raycast
    hit = raycast(agent.eye_pos, to_target_norm, distance, obstacles)
    if hit and hit.distance < distance:
        return False  # Something blocks the view

    return True

    10.3 Light Level & Stealth

    In stealth games, detection depends on the target's illumination:

    • Sample light intensity at the target's position from all light sources
    • Reduce effective detection range in dark areas: $d_{\text{eff}} = d_{\max} \cdot \text{lightLevel}$
    • Shadows create safe zones; moving through lit areas increases detection probability
    • Some systems use a detection meter that fills gradually rather than instant detection

    10.4 Auditory Perception

    Sound is typically modeled via an event system rather than physics simulation:

    • Actions generate sound events with a position, loudness, and type (footstep, gunshot, explosion)
    • Sound attenuates with distance: $\text{perceived} = \text{loudness} - 20 \log_{10}(d)$ (inverse-square law in dB)
    • Walls and obstacles reduce sound transmission (simple check: is there a clear path?)
    • AI agents "hear" events within their auditory range and can investigate the sound source
    • Different surfaces produce different footstep volumes (carpet < wood < metal)

    10.5 Fitts' Law — Humanlike Aiming

    To prevent AI from having perfect aim, apply Fitts' Law to simulate human targeting constraints:

    $$T = a + b \cdot \log_2\left(\frac{2D}{W}\right)$$

    Where $T$ is movement time, $D$ is distance to target, $W$ is target width, and $a, b$ are empirically tuned constants. This means:

    • Small, distant targets take longer to aim at
    • Large, close targets are acquired quickly
    • Add a reaction time delay before the AI begins aiming (typically 200–500ms, randomized)
    Fitts' Law: Fitts' Law is a predictive model of human motor performance stating that the time to acquire a target is logarithmically related to the ratio of distance over target width: $T = a + b \log_2(2D/W)$. In game AI, it is used to simulate humanlike aiming delays so that AI agents take longer to aim at small or distant targets, preventing unrealistically instant and perfect accuracy that would feel unfair to players.

    10.6 Gaussian Aiming Error

    Add aim inaccuracy using a 2D Gaussian distribution centered on the true target position:

    $$\text{aimPoint} = \text{target} + \mathcal{N}(0, \sigma^2) \cdot \hat{x} + \mathcal{N}(0, \sigma^2) \cdot \hat{y}$$

    The standard deviation $\sigma$ can vary by:

    • Difficulty level — easy AI has high $\sigma$, hard AI has low $\sigma$
    • Weapon type — sniper rifles have low $\sigma$, SMGs have high $\sigma$
    • Movement state — stationary AI aims better than moving AI
    • Time on target — $\sigma$ decreases the longer AI tracks a target (simulating aim refinement)
    Rubber banding: Many games secretly adjust AI accuracy based on player performance. If the player is struggling, AI misses more; if the player is dominating, AI gets more accurate. This dynamic difficulty keeps gameplay engaging without the player noticing.
    • Limit AI senses (sight range, FOV, occlusion, hearing radius) for fairness
    • Foveal/peripheral vision creates realistic detection behavior
    • Sound uses event-based systems with distance attenuation, not physics simulation
    • Fitts' Law models humanlike aim time; Gaussian noise simulates aim inaccuracy
    • Reaction time delays and dynamic difficulty prevent AI from feeling omniscient

    11. Decision Making

    11.1 Decision Trees

    The simplest decision architecture. Each internal node tests a condition; leaves are actions:

    • Binary branches — yes/no tests traverse left or right
    • Implicit AND — a path from root to leaf represents the conjunction of all conditions along that path
    • Implicit OR — multiple paths can lead to the same action, representing disjunction
    • Random nodes — replace a decision node with a random branch (e.g., 60% attack, 40% retreat) to add variety
    • Stickiness — once a random choice is made, keep it for a duration to prevent erratic behavior (e.g., random decision is re-evaluated every 5 seconds, not every frame)
    class DecisionNode:
    def __init__(self, test_fn, true_branch, false_branch):
        self.test = test_fn
        self.true_branch = true_branch
        self.false_branch = false_branch

    def evaluate(self, context):
        if self.test(context):
            return self.true_branch.evaluate(context)
        else:
            return self.false_branch.evaluate(context)

class ActionNode:
    def __init__(self, action):
        self.action = action

    def evaluate(self, context):
        return self.action

# Example: enemy decision tree
tree = DecisionNode(
    lambda ctx: ctx['health'] < 30,
    ActionNode('flee'),            # low health -> flee
    DecisionNode(
        lambda ctx: ctx['enemy_visible'],
        DecisionNode(
            lambda ctx: ctx['distance'] < 10,
            ActionNode('melee_attack'),   # close range
            ActionNode('ranged_attack')   # far range
        ),
        ActionNode('patrol')       # no enemy visible
    )
)

    11.2 Rule-Based Systems

    A set of if-then rules evaluated against a working memory (database of facts):

    • Fixed priority — rules are ordered by priority; the first matching rule fires. Simple but inflexible
    • Dynamic priority — each rule has a priority score that changes based on context. Compute all matching rules, fire the highest-priority one
    • Conflict resolution strategies — when multiple rules match: specificity (most conditions wins), recency (newest facts preferred), or random

    11.3 The Rete Algorithm

    Naive rule matching is expensive: $O(R \times F^C)$ where $R$ = rules, $F$ = facts, $C$ = conditions per rule. The Rete algorithm (Forgy, 1982) optimizes this:

    • Build a discrimination network (directed acyclic graph) from all rule conditions
    • Alpha nodes test single conditions against individual facts (intra-element tests)
    • Beta nodes join results from multiple alpha nodes (inter-element tests, variable bindings)
    • When a fact changes, only affected nodes re-evaluate — unchanged conditions are cached
    • Achieves near-constant time per fact update regardless of rule count
    Rete trade-off: The algorithm trades memory for speed. The network caches all partial matches, which can consume significant memory with many rules and facts. Modern variants (Rete II, TREAT) reduce memory usage while maintaining performance.
    Finite State Machine (FSM): A finite state machine is a computational model in which an agent occupies exactly one state at a time and transitions between states in response to events or conditions. Each state encapsulates a behavior (e.g., Patrol, Chase, Attack) with enter, update, and exit callbacks. FSMs are the most widely used decision-making architecture in game AI due to their simplicity and predictability, though they suffer from combinatorial explosion of transitions as the number of states grows.

    11.4 Finite State Machines (FSMs)

    The workhorse of game AI. An agent is always in one state; transitions are triggered by conditions:

    • Moore machine — actions are associated with states (execute action while in state)
    • Mealy machine — actions are associated with transitions (execute action during state change)
    • Each state has: enter() (setup), update() (per-frame logic), exit() (cleanup)
    class State:
    def enter(self, agent): pass
    def update(self, agent, dt): pass
    def exit(self, agent): pass

class PatrolState(State):
    def enter(self, agent):
        agent.set_animation('walk')
        agent.pick_patrol_point()

    def update(self, agent, dt):
        if agent.can_see_player():
            agent.change_state(ChaseState())
        elif agent.reached_patrol_point():
            agent.pick_patrol_point()
        agent.move_toward(agent.patrol_target, dt)

class ChaseState(State):
    def enter(self, agent):
        agent.set_animation('run')

    def update(self, agent, dt):
        if not agent.can_see_player():
            agent.change_state(SearchState())
        elif agent.in_attack_range():
            agent.change_state(AttackState())
        agent.move_toward(agent.player.pos, dt)

    11.5 Advanced FSM Variants

    • Hierarchical FSMs (HFSMs) — states contain sub-state machines. A "Combat" state might have sub-states "Aim", "Fire", "Reload". Transitions can occur at any level; exiting a parent state exits all child states
    • Stack-based FSMs — push/pop states like a call stack. Useful for interruptible behaviors: push "Investigate Noise" on top of "Patrol", then pop to resume patrol
    • Probabilistic transitions — transitions fire with a probability rather than deterministically, adding behavioral variety
    • Global states — a special state whose update runs every frame regardless of the current state, handling universal logic like "flee if health critical"
    Behavior Tree (BT): A behavior tree is a hierarchical, tree-structured control architecture where internal nodes (sequences, selectors, decorators) compose the execution logic and leaf nodes perform condition checks or actions. Each node returns one of three statuses: Success, Failure, or Running. Behavior trees offer superior modularity over FSMs because subtrees are self-contained and reusable, and adding new behaviors requires only inserting branches rather than rewiring global transition tables.

    11.6 Behavior Trees

    A modular, hierarchical decision structure that has largely replaced FSMs in modern game AI (Halo 2 popularized them):

    Node types:

    • Sequence (AND) — executes children left-to-right; succeeds only if ALL children succeed; fails on the first failure
    • Selector (OR / Fallback) — executes children left-to-right; succeeds on the FIRST success; fails only if all children fail
    • Decorator — wraps a single child to modify its behavior:
      • Inverter — flips success/failure
      • Repeater — runs child N times or until failure
      • Limiter — allows child to run at most N times total
      • Timer — succeeds/fails after a time limit
    • Leaf nodes — conditions (check world state) or actions (perform behavior), returning Success, Failure, or Running (still in progress)
    # Behavior tree node types
class Selector:
    """Try children until one succeeds (OR logic)."""
    def __init__(self, children):
        self.children = children

    def tick(self, blackboard):
        for child in self.children:
            status = child.tick(blackboard)
            if status != 'FAILURE':
                return status  # SUCCESS or RUNNING
        return 'FAILURE'

class Sequence:
    """Run children until one fails (AND logic)."""
    def __init__(self, children):
        self.children = children

    def tick(self, blackboard):
        for child in self.children:
            status = child.tick(blackboard)
            if status != 'SUCCESS':
                return status  # FAILURE or RUNNING
        return 'SUCCESS'

# Example: Guard behavior tree
guard_bt = Selector([
    Sequence([  # Combat behavior
        Condition(lambda bb: bb['enemy_visible']),
        Condition(lambda bb: bb['has_ammo']),
        Action('attack_enemy')

    11.7 Behavior Tree Concepts

    • Blackboard — a shared data store (dictionary) that nodes read from and write to. Decouples nodes from each other and from game systems
    • Semaphores — limit concurrent access to shared resources. E.g., only 3 agents can use the "flanking" behavior simultaneously; the 4th must wait or choose another action
    • Reactive behavior trees — re-evaluate from the root each tick, allowing higher-priority branches to preempt running lower-priority ones
    BT vs. FSM: Behavior trees are more modular (subtrees are reusable), easier to extend (add branches without rewiring transitions), and scale better. FSMs become unwieldy with many states due to combinatorial explosion of transitions ($n$ states → up to $n(n-1)$ transitions).

    11.8 Fuzzy Logic

    Classical boolean logic is crisp (true/false). Fuzzy logic allows degrees of truth in $[0, 1]$, better modeling vague concepts like "close" or "healthy":

    Process:

    1. Fuzzification — map crisp input values to fuzzy membership values using membership functions (triangular, trapezoidal, Gaussian): $$\mu_{\text{close}}(d) = \max\left(0, 1 - \frac{d}{d_{\max}}\right)$$
    2. Rule evaluation — apply fuzzy rules using fuzzy AND (min), OR (max), NOT (1 - x):
      IF distance IS close AND health IS high THEN aggression IS high
    3. Defuzzification — convert fuzzy output back to a crisp value. Common method is centroid: $$\text{output} = \frac{\sum_i \mu_i \cdot c_i}{\sum_i \mu_i}$$ where $c_i$ is the center of each output membership function and $\mu_i$ is its activation level
    Fuzzy Logic Pipeline: The complete fuzzy logic pipeline has three stages: (1) Fuzzification maps crisp numeric inputs to membership degrees in linguistic sets via membership functions (e.g., distance 12 maps to 0.7 "close" and 0.3 "medium"); (2) Rule evaluation applies if-then rules using fuzzy AND (min) and OR (max) operators to produce fuzzy output sets; (3) Defuzzification converts the aggregated fuzzy output back to a single crisp value, typically using the centroid method. This pipeline lets AI reason with imprecise, human-style categories rather than brittle threshold comparisons.

    11.9 Combs Method

    Standard fuzzy rule systems suffer from combinatorial explosion: $n$ inputs with $m$ values each require $m^n$ rules. The Combs method reduces this to $n \times m$ rules by decomposing multi-input rules into single-input rules combined with OR:

    • Instead of: IF A is a1 AND B is b1 THEN C is c1
    • Use: IF A is a1 THEN C is c1 OR IF B is b1 THEN C is c1
    • Results are combined by taking the max (fuzzy OR) of all applicable rule outputs
    • Trades precision for scalability — works well when inputs are somewhat independent
    • Decision trees: simple, fast evaluation via nested conditions; add random nodes for variety
    • Rule-based systems: flexible but expensive matching; Rete algorithm caches partial matches
    • FSMs: states + transitions; extend with hierarchy, stacks, probabilistic transitions
    • Behavior trees: modular sequences/selectors; blackboard for shared data; semaphores for resource limits
    • Fuzzy logic: fuzzify → evaluate rules → defuzzify; Combs method scales linearly

    12. Learning

    12.1 Decision Tree Learning — ID3 Algorithm

    The ID3 (Iterative Dichotomiser 3) algorithm automatically constructs a decision tree from labeled training examples:

    ID3 (Iterative Dichotomiser 3): ID3 is a top-down decision tree learning algorithm that recursively partitions training data by selecting the attribute with the highest information gain (greatest entropy reduction) at each node. Entropy $H(S) = -\sum p_c \log_2 p_c$ measures class impurity; the attribute that maximally reduces it becomes the splitting criterion. The recursion terminates when a node is pure (single class) or no attributes remain, yielding a tree that can classify new instances by traversing from root to leaf.

    Core concept — Entropy: measures the impurity/uncertainty in a dataset:

    $$H(S) = -\sum_{c \in C} p_c \log_2(p_c)$$

    where $p_c$ is the proportion of examples belonging to class $c$. Entropy is 0 when all examples are the same class (pure) and maximal when classes are evenly split.

    Information Gain: the reduction in entropy from splitting on attribute $A$:

    $$\text{IG}(S, A) = H(S) - \sum_{v \in \text{values}(A)} \frac{|S_v|}{|S|} H(S_v)$$

    where $S_v$ is the subset of examples where attribute $A$ has value $v$.

    Algorithm:

    1. Compute information gain for each unused attribute
    2. Select the attribute with the highest gain as the splitting criterion
    3. Create a branch for each attribute value
    4. Recurse on each subset; stop when all examples have the same class or no attributes remain
    import math
from collections import Counter

def entropy(labels):
    """Calculate entropy of a label distribution."""
    counts = Counter(labels)
    total = len(labels)
    return -sum(
        (c / total) * math.log2(c / total)
        for c in counts.values() if c > 0
    )

def info_gain(data, labels, attribute_idx):
    """Calculate information gain for splitting on an attribute."""
    total_entropy = entropy(labels)
    values = set(row[attribute_idx] for row in data)

    weighted_entropy = 0
    for v in values:
        subset_labels = [
            labels[i] for i, row in enumerate(data)
            if row[attribute_idx] == v
        ]
        weighted_entropy += (len(subset_labels) / len(labels)) * entropy(subset_labels)

    return total_entropy - weighted_entropy

def build_tree(data, labels, attributes):
    """Build decision tree using ID3."""
    # Base cases
    if len(set(labels)) == 1:
        return labels[0]  # Pure node
    if not attributes:
        return Counter(labels).most_common(1)[0][0]  # Majority vote

    # Find best attribute
    gains = {a: info_gain(data, labels, a) for a in attributes}
    best_attr = max(gains, key=gains.get)

    tree = {'attribute': best_attr, 'branches': {}}
    remaining = [a for a in attributes if a != best_attr]

    for value in set(row[best_attr] for row in data):
        subset_idx = [i for i, row in enumerate(data) if row[best_attr] == value]
        subset_data = [data[i] for i in subset_idx]
        subset_labels = [labels[i] for i in subset_idx]
        tree['branches'][value] = build_tree(subset_data, subset_labels, remaining)

    return tree

    12.2 ID4 — Live/Incremental Learning

    ID4 extends ID3 for online (incremental) learning — updating the tree as new examples arrive without rebuilding from scratch:

    • Maintain counts at each node for all attribute-value-class combinations
    • When a new example arrives, update counts along the path from root to leaf
    • Recompute information gain at each node; if a different attribute now has higher gain, restructure that subtree
    • Useful for adaptive AI that learns player patterns during gameplay

    12.3 Case-Based Reasoning (CBR)

    Learn by remembering and adapting past experiences rather than building explicit models:

    4R Cycle:

    1. Retrieve — find the most similar past case(s) from the case library. Similarity metric: $\text{sim}(\mathbf{x}, \mathbf{c}) = \sum_i w_i \cdot \text{match}_i(x_i, c_i)$ with feature weights $w_i$
    2. Reuse (Adapt) — modify the retrieved solution to fit the current problem. Adaptation rules are domain-specific
    3. Revise (Evaluate) — test the adapted solution; determine if it succeeded or failed
    4. Retain (Store) — if successful, add the new case to the library for future reference

    Efficient retrieval with KD-trees:

    • Build a KD-tree over the feature space of stored cases
    • Nearest-neighbor queries are $O(\log n)$ average case, much faster than linear search
    • Handle mixed feature types (numeric + categorical) with appropriate distance metrics
    Killer Instinct's Shadow System: Microsoft's fighting game Killer Instinct uses CBR to create shadow versions of players. The system records player behavior (attack patterns, defensive habits, combo preferences), stores them as cases, and replays adapted versions as AI opponents. Other players can fight your shadow when you're offline.

    12.4 N-Grams

    Predict the next action in a sequence based on the preceding $n-1$ actions:

    • Unigram ($n=1$): probability of each action independently $P(a)$
    • Bigram ($n=2$): probability conditioned on previous action $P(a_t | a_{t-1})$
    • Trigram ($n=3$): $P(a_t | a_{t-2}, a_{t-1})$
    • Window size trade-off — larger $n$ captures more context but requires exponentially more training data and may overfit to specific sequences
    from collections import defaultdict

class NGramPredictor:
    def __init__(self, n=3):
        self.n = n
        self.counts = defaultdict(lambda: defaultdict(int))
        self.history = []

    def observe(self, action):
        self.history.append(action)
        if len(self.history) >= self.n:
            context = tuple(self.history[-self.n:-1])
            self.counts[context][action] += 1

    def predict(self):
        if len(self.history) < self.n - 1:
            return None
        context = tuple(self.history[-(self.n-1):])
        if context not in self.counts:
            return None
        actions = self.counts[context]
        return max(actions, key=actions.get)

# Usage: predict player's next move
predictor = NGramPredictor(n=3)
for move in player_moves:
    prediction = predictor.predict()
    if prediction:
        ai_counter = counter_move(prediction)
    predictor.observe(move)

    Hierarchical N-grams: Try the longest context first; if insufficient data, back off to shorter contexts. E.g., try trigram prediction → if unseen context, fall back to bigram → fall back to unigram.

    12.5 Naive Bayes Classifier

    Classify actions using Bayes' theorem with the naive assumption that features are conditionally independent:

    $$P(C | \mathbf{x}) \propto P(C) \prod_{i=1}^{n} P(x_i | C)$$
    • $P(C)$ is the prior probability of class $C$ (how often this action occurs overall)
    • $P(x_i | C)$ is the likelihood of feature $x_i$ given class $C$ (estimated from training data)
    • Choose the class with the highest posterior probability
    • Laplace smoothing: add 1 to all counts to avoid zero probabilities for unseen feature-class combinations: $P(x_i | C) = \frac{\text{count}(x_i, C) + 1}{\text{count}(C) + |V|}$

    12.6 Exploration vs. Exploitation

    The fundamental dilemma in online learning:

    • Exploitation — use the best-known action based on current knowledge (greedy)
    • Exploration — try new or uncertain actions to potentially discover better strategies
    • Epsilon-greedy ($\varepsilon$-greedy) — with probability $\varepsilon$ choose a random action (explore), otherwise choose the best known (exploit). Common to decay $\varepsilon$ over time: $\varepsilon_t = \varepsilon_0 \cdot \gamma^t$

    12.7 Q-Learning

    A model-free reinforcement learning algorithm that learns an optimal action-value function $Q(s, a)$:

    Q-Learning: Q-learning is an off-policy, model-free reinforcement learning algorithm that learns the optimal action-value function $Q^*(s,a)$ without requiring a model of the environment. It updates Q-values using temporal difference: $Q(s,a) \leftarrow Q(s,a) + \alpha[r + \gamma \max_{a'} Q(s',a') - Q(s,a)]$, where $\alpha$ is the learning rate and $\gamma$ is the discount factor. The "off-policy" property means the agent learns the optimal policy regardless of the exploration strategy it actually follows during training.
    $$Q(s, a) \leftarrow Q(s, a) + \alpha \left[ r + \gamma \max_{a'} Q(s', a') - Q(s, a) \right]$$

    Where:

    • $s$ = current state, $a$ = action taken, $r$ = reward received
    • $s'$ = next state, $a'$ = possible actions in next state
    • $\alpha$ = learning rate (typically 0.1–0.3)
    • $\gamma$ = discount factor (0–1); higher values prioritize future rewards
    import random
from collections import defaultdict

class QLearningAgent:
    def __init__(self, actions, alpha=0.1, gamma=0.9, epsilon=0.1):
        self.Q = defaultdict(lambda: defaultdict(float))
        self.actions = actions
        self.alpha = alpha      # learning rate
        self.gamma = gamma      # discount factor
        self.epsilon = epsilon  # exploration rate

    def choose_action(self, state):
        # Epsilon-greedy exploration
        if random.random() < self.epsilon:
            return random.choice(self.actions)
        q_values = self.Q[state]
        if not q_values:
            return random.choice(self.actions)
        return max(q_values, key=q_values.get)

    def learn(self, state, action, reward, next_state):
        # Q-learning update rule
        max_next_q = max(self.Q[next_state].values()) if self.Q[next_state] else 0
        td_error = reward + self.gamma * max_next_q - self.Q[state][action]
        self.Q[state][action] += self.alpha * td_error

agent = QLearningAgent(actions=['attack', 'defend', 'heal', 'flee'])

    13. Advanced Procedural Content Generation

    13.1 PCG Overview

    Procedural Content Generation creates game content algorithmically rather than manually. Benefits include infinite variety, reduced storage requirements, and reduced authoring burden. PCG can generate terrain, levels, quests, items, music, and narrative.

    13.2 Random Numbers & Seeding

    Controlled randomness is essential for reproducible PCG:

    • Pseudorandom number generators (PRNGs) — deterministic sequences from a seed value. Same seed → same content. Essential for multiplayer synchronization and save/load
    • Seed propagation — derive sub-seeds from a master seed for independent generation streams (terrain, enemies, loot each get their own PRNG)
    • Hash functions — map coordinates directly to random values without sequential generation: $\text{value} = \text{hash}(x, y, \text{seed})$. Enables parallel generation and arbitrary-order access

    13.3 Terrain Generation

    Several approaches to heightmap terrain:

    • Perlin noise layering — combine multiple octaves of Perlin/Simplex noise at different frequencies and amplitudes: $$h(x, y) = \sum_{i=0}^{n} \text{persistence}^i \cdot \text{noise}\left(\frac{x \cdot \text{lacunarity}^i}{s}, \frac{y \cdot \text{lacunarity}^i}{s}\right)$$ Typical values: persistence $\approx 0.5$, lacunarity $\approx 2.0$
    • Fault formation — repeatedly draw random lines across the terrain; raise one side, lower the other. Many iterations produce realistic ridge patterns
    • Thermal erosion — simulate rock crumbling: if height difference between neighbors exceeds a talus angle threshold, transfer material downhill
    • Hydraulic erosion — simulate water flow: deposit rain, compute flow direction (steepest descent), erode material proportional to water volume and slope, deposit sediment where flow slows

    13.4 Object Placement

    Placing objects (trees, rocks, buildings) requires avoiding regular grid patterns while maintaining minimum spacing:

    • Halton sequence — a low-discrepancy quasi-random sequence. Generate the $n$-th point using base-$b$ digit reversal: $$H_b(n) = \sum_{i=0}^{k} d_i \cdot b^{-(i+1)}$$ where $d_i$ are the base-$b$ digits of $n$. Use bases 2 and 3 for x and y coordinates. More uniform than pseudorandom but still irregular
    • Poisson disk sampling — guarantees minimum distance $r$ between any two points:
      • Start with a random seed point
      • Generate $k$ candidates (typically 30) in the annulus $[r, 2r]$ around an active point
      • Accept a candidate if no existing point is within distance $r$ (use spatial grid for fast checking)
      • Remove active point if all candidates rejected; repeat until active list is empty
    • Density maps — modulate placement probability using a grayscale map: forests are dense in valleys, sparse on mountaintops
    Poisson Disk Sampling: Poisson disk sampling generates point distributions where no two points are closer than a minimum distance $r$, producing a "blue noise" pattern that looks natural and avoids both clumping and regular grid artifacts. Bridson's fast algorithm runs in $O(n)$ by maintaining an active list and a background grid for constant-time proximity checks. It is the preferred method for placing vegetation, rocks, and other environmental objects in games because the enforced spacing prevents overlap while the randomness avoids visible repetition.
    Horizon Zero Dawn's approach: Guerrilla Games used dithered sampling with density maps. A blue noise pattern (Poisson-disk-like) is modulated by a density field. Objects are placed where the noise value exceeds a threshold inversely proportional to desired density. This produces natural-looking distributions at any density level.

    13.5 Grammars & L-Systems

    L-systems (Lindenmayer systems) use parallel string rewriting rules to generate fractal-like structures, especially plants:

    • Axiom — initial string (e.g., "F")
    • Production rules — rewriting rules applied in parallel each iteration (e.g., F → F[+F]F[-F]F)
    • Turtle graphics interpretation:
      • F = move forward and draw
      • + = turn left by angle $\delta$
      • - = turn right by angle $\delta$
      • [ = push state (save position and heading)
      • ] = pop state (restore position and heading)
    def l_system(axiom, rules, iterations):
    """Generate L-system string by iterating production rules."""
    current = axiom
    for _ in range(iterations):
        next_str = []
        for ch in current:
            next_str.append(rules.get(ch, ch))
        current = ''.join(next_str)
    return current

# Fractal plant
plant = l_system('X', {
    'X': 'F+[[X]-X]-F[-FX]+X',
    'F': 'FF'
}, iterations=6)

# Stochastic L-system: multiple rules per symbol with probabilities
import random
def stochastic_l_system(axiom, rules, iterations):
    current = axiom
    for _ in range(iterations):
        next_str = []
        for ch in current:
            if ch in rules:
                # rules[ch] = [(prob, replacement), ...]
                r = random.random()
                cumulative = 0
                for prob, replacement in rules[ch]:
                    cumulative += prob
                    if r <= cumulative:
                        next_str.append(replacement)
                        break
            else:
                next_str.append(ch)
        current = ''.join(next_str)
    return current

    13.6 Search-Based PCG

    Treat content generation as an optimization problem:

    • Simulated annealing — start with a random level; make small random mutations. Accept improvements always; accept worse solutions with probability $P = e^{-\Delta E / T}$ where $T$ decreases over time (cooling schedule). Escapes local optima early, converges later
    • Genetic algorithms — maintain a population of candidate levels:
      1. Fitness evaluation — score each candidate on playability, difficulty, aesthetics
      2. Selection — pick parents (tournament, roulette wheel, rank-based)
      3. Crossover — combine parts of two parents to create offspring
      4. Mutation — randomly modify offspring
      5. Repeat for many generations
    • Fitness function design is the key challenge: balance multiple objectives (solvability, difficulty, diversity, aesthetic quality)
    Wave Function Collapse (WFC): Wave Function Collapse is a constraint-propagation algorithm for procedural generation that fills a grid by iteratively collapsing the cell with the lowest entropy (fewest remaining valid options) to a single tile, then propagating adjacency constraints to eliminate incompatible tiles in neighboring cells. Inspired by quantum superposition terminology, each uncollapsed cell exists in a "superposition" of all possible tiles until observation forces a choice. WFC is widely used for 2D tilemap and 3D voxel level generation because it guarantees local consistency by construction.

    13.7 Constraint Satisfaction & Wave Function Collapse

    Wave Function Collapse (WFC) generates content by propagating adjacency constraints from a tileset:

    1. Initialize — each cell starts in a superposition of all possible tiles
    2. Observe — select the cell with the lowest entropy (fewest remaining possibilities) and collapse it to a single tile (randomly weighted)
    3. Propagate — remove tiles from neighboring cells that are now incompatible with the chosen tile, based on adjacency rules
    4. Repeat until all cells are collapsed or a contradiction is reached (backtrack if so)
    import random, math
from collections import defaultdict

class WFC:
    def __init__(self, width, height, tiles, adjacency_rules):
        self.w, self.h = width, height
        self.tiles = tiles
        self.rules = adjacency_rules  # {(tile, direction): set of valid neighbors}
        # Each cell starts with all tiles possible
        self.grid = [[set(tiles) for _ in range(width)] for _ in range(height)]

    def entropy(self, x, y):
        n = len(self.grid[y][x])
        return math.log2(n) if n > 1 else 0

    def find_min_entropy(self):
        min_e, best = float('inf'), None
        for y in range(self.h):
            for x in range(self.w):
                e = self.entropy(x, y)
                if 0 < e < min_e:
                    min_e = e
                    best = (x, y)
        return best

    def collapse(self, x, y):
        tile = random.choice(list(self.grid[y][x]))
        self.grid[y][x] = {tile}
        self.propagate(x, y)

    def propagate(self, x, y):
        stack = [(x, y)]
        while stack:
            cx, cy = stack.pop()
            for dx, dy, direction in [(0,-1,'N'),(1,0,'E'),(0,1,'S'),(-1,0,'W')]:
                nx, ny = cx + dx, cy + dy
                if 0 <= nx < self.w and 0 <= ny < self.h:
                    valid = set()
                    for tile in self.grid[cy][cx]:
                        valid |= self.rules.get((tile, direction), set())
                    before = len(self.grid[ny][nx])
                    self.grid[ny][nx] &= valid
                    if len(self.grid[ny][nx]) < before:
                        stack.append((nx, ny))

    def generate(self):
        while True:
            cell = self.find_min_entropy()
            if cell is None:
                break
            self.collapse(*cell)

    Adapt generated content to individual players:

    • Flow theory (Csikszentmihalyi) — optimal experience occurs when challenge matches skill. Too easy → boredom; too hard → frustration. PCG can dynamically adjust difficulty to maintain flow
    • Drama management — control narrative pacing by adjusting encounter frequency, loot drops, and story beats based on player engagement metrics

    14.3 Chase Camera with Spring Damping

    A spring-damped chase camera avoids jerky movement by treating the camera as a mass-spring system:

    $$\mathbf{a}_{\text{cam}} = k_s (\mathbf{p}_{\text{ideal}} - \mathbf{p}_{\text{cam}}) - k_d \cdot \mathbf{v}_{\text{cam}}$$

    Where $k_s$ is the spring stiffness (higher = snappier tracking), $k_d$ is the damping coefficient (higher = less oscillation), and $\mathbf{p}_{\text{ideal}}$ is the desired camera position behind the player.

    Spring-Damped Chase Camera: A spring-damped chase camera models the camera as a mass connected to the player by a virtual spring and dashpot, governed by $\mathbf{a} = k_s(\mathbf{p}_{\text{ideal}} - \mathbf{p}_{\text{cam}}) - k_d \mathbf{v}_{\text{cam}}$. The spring constant $k_s$ controls responsiveness (how quickly the camera catches up) while the damping constant $k_d$ suppresses oscillation. This second-order system produces smooth, physically plausible camera motion that avoids both the rigidity of hard-attached cameras and the sluggishness of pure interpolation. 
    class ChaseCamera:
    def __init__(self, offset, stiffness=5.0, damping=3.0):
        self.offset = offset       # Desired offset from target (e.g., [0, 5, -10])
        self.stiffness = stiffness
        self.damping = damping
        self.velocity = Vector3(0, 0, 0)

    def update(self, target_pos, target_forward, dt):
        # Compute ideal position behind target
        ideal = target_pos + target_forward * self.offset.z + Vector3(0, self.offset.y, 0)

        # Spring-damper acceleration
        displacement = ideal - self.position
        accel = self.stiffness * displacement - self.damping * self.velocity

        # Integrate
        self.velocity += accel * dt
        self.position += self.velocity * dt

        # Always look at target
        self.look_at = target_pos + Vector3(0, 1.5, 0)  # Slightly above center

    • Constraint-based planners balance hard requirements (visibility) with soft preferences (framing)

    15. Minimax & Monte Carlo Tree Search

    15.1 Zero-Sum Games

    A zero-sum game is one where one player's gain is exactly the other's loss. Total utility sums to zero. Examples: chess, checkers, Go, tic-tac-toe. Formally, for payoffs $u_1$ and $u_2$: $u_1 + u_2 = 0$.

    15.2 Minimax Algorithm

    Optimal play in zero-sum games: the maximizing player chooses moves that maximize the evaluation; the minimizing player chooses moves that minimize it:

    $$\text{minimax}(s) = \begin{cases} \text{eval}(s) & \text{if } s \text{ is terminal or depth limit} \\ \max_{a} \text{minimax}(\text{result}(s, a)) & \text{if MAX's turn} \\ \min_{a} \text{minimax}(\text{result}(s, a)) & \text{if MIN's turn} \end{cases}$$ 
    def minimax(state, depth, is_maximizing):
    if depth == 0 or state.is_terminal():
        return state.evaluate()

    if is_maximizing:
        best = float('-inf')
        for move in state.get_moves():
            state.make_move(move)
            val = minimax(state, depth - 1, False)
            state.undo_move(move)
            best = max(best, val)
        return best
    else:
        best = float('inf')
        for move in state.get_moves():
            state.make_move(move)
            val = minimax(state, depth - 1, True)
            state.undo_move(move)
            best = min(best, val)
        return best

    Complexity: $O(b^d)$ where $b$ is the branching factor and $d$ is the depth. Chess has $b \approx 35$ and $d \approx 80$ for a full game, making exhaustive search impossible.

    15.3 Alpha-Beta Pruning

    Prune branches that cannot affect the final decision, reducing the effective branching factor:

    • $\alpha$ = best value MAX can guarantee so far (lower bound)
    • $\beta$ = best value MIN can guarantee so far (upper bound)
    • Prune when $\alpha \geq \beta$ (the current branch cannot produce a better result for the player whose turn it is)
    def alpha_beta(state, depth, alpha, beta, is_maximizing):
    if depth == 0 or state.is_terminal():
        return state.evaluate()

    if is_maximizing:
        value = float('-inf')
        for move in state.get_moves():
            state.make_move(move)
            value = max(value, alpha_beta(state, depth - 1, alpha, beta, False))
            state.undo_move(move)
            alpha = max(alpha, value)
            if alpha >= beta:
                break  # Beta cutoff - MIN won't allow this
        return value
    else:
        value = float('inf')
        for move in state.get_moves():
            state.make_move(move)
            value = min(value, alpha_beta(state, depth - 1, alpha, beta, True))
            state.undo_move(move)
            beta = min(beta, value)
            if alpha >= beta:
                break  # Alpha cutoff - MAX won't allow this
        return value

    15.4 Evaluation Functions

    When we can't search to terminal states, an evaluation function estimates position quality:

    • Material counting — weighted sum of piece values (chess: pawn=1, knight/bishop=3, rook=5, queen=9)
    • Positional features — piece mobility, center control, king safety, pawn structure
    • Linear combination: $\text{eval}(s) = \sum_i w_i \cdot f_i(s)$ where $f_i$ are features and $w_i$ are learned weights
    • Neural network evaluators — modern engines (AlphaZero, Stockfish NNUE) use neural networks trained on millions of positions

    15.5 Iterative Deepening

    Search to depth 1, then depth 2, then depth 3, etc., until time runs out:

    • Always has a complete result from the previous depth to fall back on (anytime algorithm)
    • Previous iterations provide move ordering for alpha-beta in deeper searches
    • Overhead is small: most nodes are at the deepest level, so re-searching shallow levels costs little
    • Combined with a transposition table (hash table of previously evaluated positions) to avoid redundant work

    15.6 Expectiminimax

    Extends minimax for games with chance (dice, card draws). Add chance nodes that compute expected value:

    $$\text{expecti}(s) = \begin{cases} \max_{a} \text{expecti}(\text{result}(s, a)) & \text{MAX's turn} \\ \min_{a} \text{expecti}(\text{result}(s, a)) & \text{MIN's turn} \\ \sum_{o} P(o) \cdot \text{expecti}(\text{result}(s, o)) & \text{chance node} \end{cases}$$

    Example: Backgammon has dice rolls between moves. The chance node averages over all 21 possible dice outcomes.

    Monte Carlo Tree Search (MCTS): MCTS is an anytime, simulation-based search algorithm that builds a search tree incrementally through repeated iterations of four steps: selection (traverse the tree using a bandit policy), expansion (add a child node), simulation (play out random moves to a terminal state), and backpropagation (update win/visit statistics along the path). Unlike minimax, MCTS requires no evaluation function because terminal game outcomes from random rollouts serve as value estimates, making it applicable to games where crafting heuristics is infeasible.

    15.7 Monte Carlo Tree Search (MCTS)

    A simulation-based search that doesn't need an evaluation function. Four phases repeated iteratively:

    1. Selection — traverse the tree from root using a selection policy (UCT) until reaching a node with unexplored children
    2. Expansion — add one (or more) child nodes for unexplored moves
    3. Simulation (Rollout) — play random moves from the new node until the game ends, obtaining a win/loss result
    4. Backpropagation — update visit counts and win counts for all nodes along the path from the new node back to the root

    15.8 UCT Formula

    The Upper Confidence Bound for Trees (UCT) balances exploitation (high win rate) with exploration (low visit count):

    $$\text{UCT}(v) = \frac{w_v}{n_v} + C \sqrt{\frac{\ln N_p}{n_v}}$$

    Where:

    • $w_v$ = number of wins after playing through node $v$
    • $n_v$ = number of visits to node $v$
    • $N_p$ = number of visits to the parent node
    • $C$ = exploration constant (typically $\sqrt{2}$); higher $C$ favors exploration
    UCT Exploitation-Exploration Balance: The UCT formula $\frac{w}{n} + C\sqrt{\frac{\ln N}{n}}$ explicitly balances two competing objectives. The first term (exploitation) favors nodes with high win rates, directing search toward promising moves. The second term (exploration) favors rarely-visited nodes, ensuring no branch is permanently ignored. The constant $C$ controls this trade-off: lower $C$ produces deeper, more focused search while higher $C$ produces broader, more exploratory search. UCT is provably optimal in the limit, converging to the minimax value as iterations increase. 
    import math, random

class MCTSNode:
    def __init__(self, state, parent=None, move=None):
        self.state = state
        self.parent = parent
        self.move = move
        self.children = []
        self.wins = 0
        self.visits = 0
        self.untried_moves = state.get_moves()

    def uct_value(self, c=1.414):
        if self.visits == 0:
            return float('inf')
        exploitation = self.wins / self.visits
        exploration = c * math.sqrt(math.log(self.parent.visits) / self.visits)
        return exploitation + exploration

    def best_child(self, c=1.414):
        return max(self.children, key=lambda n: n.uct_value(c))

def mcts(root_state, iterations=1000):
    root = MCTSNode(root_state)

    for _ in range(iterations):
        node = root
        state = root_state.copy()

        # 1. Selection - traverse tree using UCT
        while not node.untried_moves and node.children:
            node = node.best_child()
            state.make_move(node.move)

        # 2. Expansion - add one child
        if node.untried_moves:
            move = random.choice(node.untried_moves)
            node.untried_moves.remove(move)
            state.make_move(move)
            child = MCTSNode(state.copy(), parent=node, move=move)
            node.children.append(child)
            node = child

        # 3. Simulation - random rollout
        sim_state = state.copy()
        while not sim_state.is_terminal():
            sim_state.make_move(random.choice(sim_state.get_moves()))
        result = sim_state.get_result()  # 1 = win, 0 = loss

        # 4. Backpropagation - update stats
        while node is not None:
            node.visits += 1
            node.wins += result
            node = node.parent
            result = 1 - result  # Flip perspective

    # Return most-visited child's move
    return max(root.children, key=lambda c: c.visits).move

    15.9 MCTS Enhancements

    • Heavy playouts — use heuristic or learned policies instead of uniform random moves during rollouts for more informative simulations
    • RAVE (Rapid Action Value Estimation) — share statistics between nodes that play the same move, even in different positions. Accelerates early convergence
    • Progressive widening — for games with very large action spaces, gradually increase the number of children considered as visit count grows
    • Parallelization — root parallelization (multiple trees, merge results), leaf parallelization (parallel rollouts from same leaf), or virtual loss (pretend the node lost to encourage other threads to explore elsewhere)
    • Minimax: recursively maximize/minimize for two-player zero-sum games; $O(b^d)$ complexity
    • Alpha-beta pruning reduces to $O(b^{d/2})$ with good move ordering
    • Evaluation functions estimate position quality when full search is infeasible
    • Iterative deepening provides anytime behavior and improves move ordering
    • Expectiminimax handles chance nodes by computing expected values
    • MCTS uses selection (UCT), expansion, rollout, backpropagation — no evaluation function needed
    • UCT balances exploitation (win rate) with exploration ($C\sqrt{\ln N / n}$)
